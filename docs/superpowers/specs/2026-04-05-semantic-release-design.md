# Semantic Release Design

**Date:** 2026-04-05
**Status:** Approved

## Goal

Automate versioning, changelog generation, and PyPI publishing using `python-semantic-release`, triggered on every merge to `master` with conventional commit enforcement on PRs.

## Background

Currently `setup.py` hardcodes `version="1.0.1"` and releases are triggered manually by pushing a `publish-pypi-*` tag. Commit messages are inconsistent. This design replaces that with fully automated, convention-driven releases.

## Changes

### 1. Replace `setup.py` with `pyproject.toml`

Delete `setup.py` and migrate all package metadata to a `[project]` table in `pyproject.toml`. The build backend remains `setuptools`. Version starts at `1.0.1` and is owned by PSR from that point forward.

`pyproject.toml` will contain:
- `[build-system]` — existing setuptools config (unchanged)
- `[project]` — name, version, description, authors, license, requires-python (`>=3.10`), dependencies, classifiers, URLs
- `[tool.semantic_release]` — PSR configuration (see below)

### 2. PSR Configuration (`[tool.semantic_release]`)

```toml
[tool.semantic_release]
version_toml = ["pyproject.toml:project.version"]
branch = "master"
major_on_zero = false
changelog_file = "CHANGELOG.md"
build_command = "pip install build && python -m build --sdist --wheel --outdir dist/"

[tool.semantic_release.publish]
upload_to_pypi = true
upload_to_release = true

[tool.semantic_release.remote.token]
env = "GH_TOKEN"
```

### 3. Release Workflow (`.github/workflows/release.yaml`)

Triggers on `push` to `master`. Steps:
1. `actions/checkout@v4` with `fetch-depth: 0` (full history required by PSR)
2. `actions/setup-python@v5` with Python 3.13
3. Install `python-semantic-release`
4. `semantic-release version` — determines next version from commits, bumps `pyproject.toml`, commits, tags
5. `semantic-release publish` — builds and uploads to PyPI

Uses `GH_TOKEN` (set to `secrets.GITHUB_TOKEN`) and `PYPI_API_KEY` secret. The workflow must declare `permissions: contents: write` and `permissions: id-token: write` so the GitHub token can push the version bump commit and tag back to master.

The existing `publish-to-pypi.yaml` is deleted.

### 4. Commit Enforcement (`.github/workflows/conventional-commits.yaml`)

Triggers on `pull_request` (types: `opened`, `synchronised`, `edited`, `reopened`). Uses `amannn/action-semantic-pull-request` to validate the PR title follows angular conventional commit format. A single-commit PR also validates the commit message directly.

Valid types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`.

Breaking changes: append `!` after type (e.g. `feat!: remove deprecated API`) or include `BREAKING CHANGE:` in the commit body.

### 5. `CLAUDE.md` Update

Add a section documenting the conventional commit format and version bump rules:
- `feat:` → minor bump
- `fix:` → patch bump
- `feat!:` / `BREAKING CHANGE:` → major bump
- `chore:`, `ci:`, `docs:`, etc. → no release

### 6. `CHANGELOG.md`

Auto-generated by PSR as part of the release step. Committed to `master` in the same release commit. No manual maintenance required.

## Out of Scope

- Migrating to `uv` or other build tools
- Pre-release versions (`alpha`, `beta`, `rc`)
- GitHub Releases with release notes (PSR supports this but not needed now)

## Success Criteria

- Merging a `feat:` PR to master automatically bumps minor version, publishes to PyPI, and updates CHANGELOG.md
- Merging a `fix:` PR bumps patch version
- Non-conventional PRs are blocked from merging by the commit check
- `setup.py` no longer exists in the repo
